## Project Overview

This repository is a The Binding of Isaac mod named "Conch's Blessing". The Steam install directory is named "Rebirth", but that directory name does not establish the supported executable/API baseline. The repository currently does not pin a minimum Repentance or Repentance+ build, so preserve existing capability checks and fallbacks unless the support policy is explicitly changed.

Runtime load order is:

1. `main.lua` requires `scripts/conch_blessing_core.lua`.
2. Core hard-gates initialization on StatsAPI, creates the IsaacScript Common-upgraded `ConchBlessing` mod, initializes SaveManager/config/MCM/HiddenItemManager, and attaches the external StatsAPI tables.
3. Core requires `scripts/rooms/init.lua` after shared managers and StatsAPI attachment. That loader owns non-item room systems and currently initializes Appraisal Certificate's native-room manager plus its optional StageAPI Atropos return-door integration.
4. Core requires `scripts/conch_blessing_items.lua`; the item loader then requires `scripts/eid_language.lua`, `scripts/callback_manager.lua`, and the item behavior modules, and registers ItemData callbacks after all behavior modules have loaded but before the first game-start dispatch.
5. Core finally loads `scripts/conch_blessing_upgrade.lua` and `scripts/conch_blessing_highlight.lua`.

Item behavior usually lives under `scripts/items/collectibles`, `scripts/items/trinkets`, or `scripts/items/familiars`. `scripts/rooms/init.lua` is a narrow loader for the implemented gallery subsystem; the repository still has no generic monster/entity/stage registry or authored room-pack loader. Do not assume non-item content is handled by `ConchBlessing.ItemData`.

The mod depends on external runtime APIs:

- StatsAPI is a hard runtime dependency; without `_G.StatsAPI.stats.unifiedMultipliers`, core initialization stops.
- Magic Conch supplies the upgrade callback API used by `scripts/conch_blessing_upgrade.lua` and is a required Workshop dependency.
- Mod Config Menu and EID integrations are optional at runtime and must remain guarded when their globals or methods are unavailable.
- EID descriptions and icons are generated from item data in `scripts/conch_blessing_items.lua` and `scripts/eid_language.lua`.
- REPENTOGON is not a blanket hard dependency, but Appraisal Certificate requires its dynamic native-room placement, post-collision evidence, and reversible smelt-inventory APIs. That active must fail closed before discharge, payment, inventory mutation, or transition when those capabilities are absent or incompatible.
- StageAPI 2.33 is required only when Appraisal is activated with the Atropos synergy, where it supplies the entrance room's custom return door. The rest of Appraisal's rooms, minimap, stock, acquisition, and transitions are native/REPENTOGON-owned.

Core subsystem map:

- `scripts/callback_manager.lua` owns callback-key to `ModCallbacks` registration.
- `scripts/template.lua` owns upgrade morph visuals before/after item conversion.
- `scripts/conch_blessing_upgrade.lua` owns validated Magic Conch origin/flag mappings, callback registration, and per-pedestal conversion transactions.
- `scripts/conch_blessing_config.lua` owns language/debug/spawn settings.
- `scripts/conch_blessing_mcm.lua` owns MCM UI and setting persistence.
- `scripts/lib/weapon_attack_tracker.lua` owns reusable direct-player weapon-trigger counting and direction resolution; item modules own and reset their own counter state.
- `scripts/lib/damage_provenance.lua` owns actual damage-source/player ownership resolution, namespaced per-proc causal-lineage marking/merging/inheritance, and exact attack-instance proc claims for damage-triggered effects.
- `scripts/rooms/native_gallery_rooms.lua` registers Appraisal's empty HOME room configs and owns deterministic native dimension-2 graph placement, serializable room manifests, runtime descriptor rebinding, stock-slot geometry, and vanilla-minimap visibility.
- `scripts/rooms/gallery_manager.lua` owns Appraisal's floor-scoped frozen catalog/closed-room ledger, exact manager-stock identity, per-visit entry/acquisition/smelt transactions, optional StageAPI Atropos return door, and native return transitions. Appraisal delegates to this manager; Atropos keeps its separate vanilla Death Certificate behavior and must ignore manager-owned Appraisal rooms.
- External `StatsAPI.stats` owns unified stat multipliers and the multiplier display; there is no local `scripts/lib/stats.lua`.
- `ConchBlessing` is the IsaacScript Common-upgraded mod object. `ConchBlessing.originalMod` is the raw `RegisterMod` reference retained for support-library and legacy call sites.

## Files To Treat Carefully

- `scripts/lib/isaacscript-common.lua` is vendored and very large. Do not edit it unless the task explicitly requires a vendor patch.
- `scripts/lib/save_manager.lua` and `scripts/lib/hidden_item_manager.lua` are shared support libraries. Prefer changing call sites first.
- `scripts/items/dragon copy.lua` is a legacy, unrequired duplicate and is not the runtime Dragon implementation; `scripts/items/collectibles/dragon.lua` is the loaded source of truth. Do not implement or validate runtime behavior against the copy.
- `content/items.xml`, `content/itempools.xml`, `content/entities2.xml`, `content/gfx/death_items.anm2`, and `content/gfx/death_items.png` are generator-owned outputs. Do not hand-edit them unless the generation workflow is intentionally being changed.
- `generate_xml.py` currently rebuilds all of `content/entities2.xml` from familiar entries in `ConchBlessing.ItemData`, forces entity type `3`, and does not preserve manual non-familiar entries. Before adding a monster, NPC, custom effect, or other non-familiar entity, first extend the source schema/generator or replace it with a documented merge/manual source-of-truth workflow.
- `metadata.xml` is hand-authored release metadata. Do not conflate it with generated content or change its version unless the version rules below authorize it.
- Text files are UTF-8. On Windows, explicitly read and write UTF-8 when a tool's default encoding is ambiguous; apparent mojibake can be a terminal decoding issue. Avoid broad encoding normalization or translation rewrites outside a text/encoding task.
- Treat files under `../../extracted_resources` and other installed mods as read-only references. Never copy another mod's implementation or assets into this repository.

## Code Style

- Keep Lua changes small and local to the item or subsystem being modified.
- Treat user examples as acceptance cases, not implementation conditions. Implement the underlying gameplay event or state transition once so it applies consistently to current and future content; do not enumerate known items, protections, entities, or room cases to simulate the requested result.
- Prefer the engine's canonical semantic operation (for example real death, acquisition, spawn, or room transition) over manipulating adjacent side effects until the example appears to work. If no canonical API exists, introduce one centralized, capability-checked abstraction with a documented fallback and limitation instead of scattered special-case branches.
- Prefer existing patterns: add item definitions to `ConchBlessing.ItemData`, then register behavior through the `callbacks` table.
- Keep `ConchBlessing.ItemData` grouped in this order: collectible items (`passive` and `active`) first, familiar items second, trinkets last. Append new items to the end of their group because generated `content/items.xml` IDs follow this registry order.
- Generated XML follows the registry's textual order, but runtime item scripts are loaded with `pairs`; do not make runtime initialization depend on ItemData iteration order.
- Keep one global table per item namespace, for example `ConchBlessing.dragon` or `ConchBlessing.chronus`.
- Use `local` helpers and constants inside item files. Do not add globals unless they are part of the existing `ConchBlessing.*` API.
- Use `ConchBlessing.printDebug` for optional logs and `ConchBlessing.printError` for real failures. Avoid unconditional `Isaac.ConsoleOutput` in normal paths.
- When loading a persisted debug toggle, apply the authoritative saved settings before emitting optional diagnostics. Never let unordered `pairs` iteration temporarily enable output or leak messages when the final saved `debugMode` is off; diagnostic state must not affect gameplay or persistence.
- For a condition- or currency-gated active whose registry contract is `maxcharges = 0`, successful use must still return `Discharge = false`; its own transaction decides whether the use succeeds. Keep engine use-item animation separate from a semantic room-transition animation so `ShowAnim` does not overlap the transition.

## Non-Item Content Expansion

- `scripts/rooms/init.lua` is the explicit non-item room loader. It is required by `scripts/conch_blessing_core.lua` after shared managers are ready and before ItemData behavior loads. It currently loads only Appraisal's native-room/gallery manager; there is still no generic monster, entity, stage, or authored-room-pack registry.
- New room systems should register through this loader or a documented registry beneath it. Do not hide monsters, rooms, or stages inside `ConchBlessing.ItemData` merely to reuse the item callback manager.
- Prefer future behavior paths such as `scripts/entities/<category>/<key>.lua`, `scripts/rooms/<key>.lua`, and `scripts/stages/<key>.lua`. Create only the paths needed by the implemented subsystem and keep one clear Lua namespace/registry entry per content key.
- Namespace new non-item runtime assets under a mod-specific path such as `resources/gfx/conch_blessing/...` to reduce cross-mod resource collisions. Existing assets do not need to be moved solely for consistency.

### Custom Entities And Monsters

- Define each custom entity's XML metadata and Lua identity from one documented source of truth. Keep the name and type/variant/subtype consistent across `entities2.xml`, Lua, and room layouts; avoid scattering unexplained numeric IDs. Prefix new XML names and runtime data keys with a Conch Blessing-specific namespace to avoid cross-mod collisions.
- Resolve custom entity IDs with `Isaac.GetEntityTypeByName` and `Isaac.GetEntityVariantByName` after XML is loaded, cache the results, and call `ConchBlessing.printError` and skip registration if lookup returns an invalid value.
- `content/entities2.xml` declares entity identity, physics, health/tags, and its `anm2path`; the ANM2 and spritesheets are runtime assets under `resources/gfx`. With the current root `anm2root="gfx/"`, entity `anm2path` values are relative to `resources/gfx`, and ANM2 spritesheet paths are relative to the ANM2/runtime resource rules.
- Register NPC callbacks when the entity subsystem loads, not through the delayed ItemData callback pass. Use the callback's supported type parameter, then return early on variant and subtype before doing AI, room scans, target selection, or rendering work.
- Model AI as explicit semantic states. Prefer `Sprite:IsEventTriggered`, `Sprite:IsFinished`, pathfinding/target state, and collision events over hardcoded animation frame numbers or global frame counts.
- Keep transient entity state in `Entity:GetData()` under a sufficiently unique key such as `__ConchBlessing...`, or in a room-scoped runtime table. `GetData()` is shared by every mod. Persist only serializable state that must survive a transition through SaveManager, and clear cached entity references on removal, room change, new level, game exit, and mod unload as applicable.
- Use seeded entity/room RNG for attacks, drops, and room-authored random choices. Prefer `Game():Spawn` with a deterministic nonzero seed when spawn identity matters. Test friendly/charmed/frozen states and multiplayer targeting when the entity can interact with them.
- Treat kill, completed NPC death, entity removal, and room-unload cleanup as different lifecycle events. `MC_POST_ENTITY_REMOVE` alone is not proof that an entity died.

### Rooms, Room Packs, And Stages

- Distinguish runtime room behavior (`Room`, `Level`, `RoomDescriptor`, callbacks) from authored room layouts (`.stb`). Create Repentance-compatible layouts with Basement Renovator or another verified compatible editor; do not hand-edit binary STB files.
- Vanilla-pool room packs belong under `content/rooms` and must use the stage filenames/formats expected by the game/editor. Verify stage, room type, shape, doors, difficulty/weight, and custom entity IDs against extracted vanilla rooms before committing. `resources/rooms` replaces vanilla room sets and is not the default additive path.
- Use `MC_PRE_ROOM_ENTITY_SPAWN` only for layout-spawn substitution, `MC_POST_NEW_ROOM` for room-entry state, and `MC_PRE_SPAWN_CLEAN_AWARD` for pre-award behavior. Read their return contracts before implementation; the local CallbackManager alias `postRoomClear` still maps to the pre-award callback.
- Key room state through `SaveManager.GetRoomSave(..., listIndex)` using the relevant `RoomDescriptor.ListIndex`; do not persist descriptor userdata or rely only on `Level:GetCurrentRoomIndex()`. Choose normal versus `noHourglass` storage deliberately, and account for first visit, room shape, dimension, revisit, continue, Glowing Hourglass, Curse of the Maze, and special rooms outside the normal map when relevant.
- The base `MC_LEVEL_GENERATOR` callback is documented as inactive/broken and must not be used to build floors. Appraisal explicitly uses REPENTOGON `RoomConfig.AddRooms` plus `Level:TryPlaceRoom` / `TryPlaceRoomAtDoor`; that decision does not automatically authorize REPENTOGON or StageAPI for unrelated stages, vanilla room generation, monsters, or bosses.
- REPENTOGON `RoomConfig.AddRooms` targets one concrete `RoomConfigSet`: pass `Mode = 0` for Normal and `Mode = 1` for Greed, never the `-1` wildcard accepted by lookup APIs. Register Appraisal's stable config sequence only for the current run mode, cache success/failure per mode, and select that mode's userdata for placement. HOME has no usable Greed room set in the installed provider; a rejected Greed registration must fail Greed activation closed without invalidating a successful Normal registration. Within one mode, a missing/rejected returned config is a permanent partial-registration failure for that Lua state because retrying can duplicate earlier database entries. In installed 1.1.2d, Lua-room conversion derives dimensions from `SHAPE` and accepts valid door `SLOT` values without `GRIDX`/`GRIDY`; do not invent stricter mandatory fields than the provider actually enforces.
- In the installed REPENTOGON/provider build, a dynamically added HOME `ROOMSHAPE_IH` config converts successfully but uses a 13x3 shape dimension against the runtime's special horizontal-room wall template and emits 19 `make_wall index out of bounds` assertions on every entry. Appraisal's entrance is therefore a verified 1x1 with LEFT0/RIGHT0; adding Lua `WIDTH`/`HEIGHT` or door `GRIDX`/`GRIDY` cannot repair this because the provider ignores those values when `SHAPE` and a valid `SLOT` are present. Do not reintroduce IH without re-testing the exact provider/build assertion-free.
- Treat both `nil` and boolean `false` from REPENTOGON `Level:TryPlaceRoom` / `TryPlaceRoomAtDoor` as placement failure; neither is a `RoomDescriptor`, and no manifest/save step may consume it. Runtime-added RoomConfig registrations are process-lifetime state. Appraisal may survive a Lua reload only by resolving every stable HOME/type/variant entry through `RoomConfig.GetRoomByStageTypeAndVariant` in the exact mode and validating its type, variant, subtype, name, shape, difficulty, and door mask before adoption; a partial or mismatched set must fail closed. Any config schema/identity/order change that cannot pass that exact adoption requires a full executable restart. Generated ItemConfig changes always require a full restart.
- Appraisal registers empty HOME subtype-34 configs for a fixed-grid-80 1x1 horizontal entrance and 1x1, 1x2, 2x1, and 2x2 stock rooms. Stock capacities are deliberate layout/performance budgets—currently 28, 52, 60, and 76—not the geometric maximum; keep a bounded per-room maximum so a large catalog expands into multiple rooms instead of one entity-dense room. New floor graphs currently use the centralized `STOCK_LAYOUT_WEIGHTS` policy `100/0/0/0` for 1x1/1x2/2x1/2x2: every new stock room is one-cell 1x1, and a zero-weight layout must also be absent from placement fallbacks. Keep the larger registered layouts and their validation so an already-frozen mixed-shape floor can restore safely and a future balance change can be made in the one policy table without changing stable room identities. The graph branches in native dimension 2; whenever the catalog requires at least three stock rooms, the first stock room must place two distinct child branches before ordinary frontier growth. Keep the entrance's LEFT0 free for Atropos's return door. Stock positions use dense adjacent-grid patterns derived from the vanilla `35.home.stb` Death Certificate layouts, leave every possible door-aligned cross axis clear, and are ordered in 180-degree pairs so a partially filled final room remains balanced. Do not restore sparse every-other-cell spacing, fill a door axis, or add these runtime-generated layouts under `content/rooms`/vanilla stage resources.
- Every placed room has a unique stable manifest key separate from its reusable layout key. Persist only serializable manifest fields and rebind by `RoomDescriptor.ListIndex`, dimension, config variant/shape/subtype/name, grid identity, and spawn seed; never persist descriptor userdata or treat a shape name as room identity.
- Appraisal sets each owned descriptor to clear and standard minimap visibility (`DisplayFlags | 5`), then calls `Level:UpdateVisibility()`. The resulting native descriptor graph is the source for the game's default minimap; do not add a custom HUD or MiniMAPI representation for this subsystem.
- Native dimension 2 is the engine's shared Death Certificate dimension, not a fourth isolated dimension. Before constructing or reusing a graph, enumerate and validate the whole dimension. If it is already occupied before construction, contains a foreign room, or no longer matches the saved manifest, fail closed before payment, held-trinket mutation, or transition. The entrance must bind to both `GridIndex` and `SafeGridIndex` 80; another anchor is not a fallback because the vanilla Death Certificate effect only inspects and targets that grid.
- REPENTOGON exposes no removal transaction for placed native rooms. Build the graph at most once per floor, persist each partial placement before continuing, reuse that graph for every paid visit on the floor, and reset its catalog/graph/closed-room ledger for the next floor. `MC_POST_NEW_LEVEL` must not discard an unresolved entry or pickup/Smelter journal: durably mark the old floor ended, finish rollback or exact reward settlement without trying to return to a destroyed origin, then remove the old session before allowing a new-floor build. A partial or ambiguous build remains failed closed for its floor; an interrupted build with no placed dimension-2 descriptor may be discarded safely.
- Appraisal freezes the sorted registry of every non-hidden trinket config once per floor and assigns it to fixed per-room slots. This is a complete registry showcase, so `ItemConfig_Item:IsAvailable()` and the current save's unlock state must not remove a valid config; exclude only missing and `Hidden == true` configs, and do not create a separate golden copy. Manager stock identity requires the owned graph room, manifest slot, catalog ID, and saved `InitSeed`; player-dropped or unrelated trinkets in the same room must remain untouched. `EntityFlag.FLAG_PERSISTENT` means an entity follows into every room, so it must never be used to preserve room-owned stock. Let the native room save respawn its pickups on reentry, rebind them by saved identity, and index the current room's pickups once per preparation/removal pass instead of calling `Isaac.FindByType` once per slot.
- Appraisal's coverage diagnostics must go through `ConchBlessing.printDebug`, which already writes both the log and console only while the saved MCM debug mode is enabled. At catalog freeze, report the scanned range and exact included/hidden/missing/unavailable-but-included IDs; after graph construction or restore, compare the frozen catalog to the current live non-hidden registry before reporting overall success, then report catalog-to-slot coverage, duplicates, room layouts, and the one-cell-policy result. An old 128-entry frozen graph may report internal 128/128 consistency but must report registry/overall failure against a live 210-entry registry, never a misleading complete `PASS`. After each stock room is physically initialized or rebound, report exact source-seed/subtype verification and cumulative visited-room coverage. Do not emit these success diagnostics when debug mode is off, do not use them as gameplay state, and keep real fail-closed errors on `printError` regardless of debug mode.
- Glowing Hourglass is deliberately blocked only while physically inside a manager-owned Appraisal room. Manual main-active and exactly identifiable pocket/copy-card uses are suppressed at `MC_INPUT_ACTION` before consumption; `MC_PRE_USE_ITEM` remains the programmatic/custom-input safety barrier. Do not generalize this suppression to native Death Certificate or other room systems.
- Do not make REPENTOGON mandatory for the whole mod or replace vanilla stage resources without an explicit compatibility/dependency decision. `resources/stages.xml` replaces vanilla data and `content/stages.xml` has no effect, so stages require a dedicated compatibility plan.
- Do not conflate entity type/variant/subtype, room type/variant/subtype, `LevelStage`, `StageType`, STB stage IDs, or entity `bossID`; they are different identifier domains even when numeric values overlap.
- Vanilla grid entities can be spawned or changed, but a brand-new grid-entity type requires an explicitly adopted extender/framework mechanism or a normal entity/effect emulation.
- Custom bosses require more than an NPC flag: room layouts, boss-pool/framework integration, portraits/transition assets, doors/clear behavior, and bestiary behavior must be designed and verified together. Route exact fields and APIs through `ISAAC_DOCS_REFERENCE.md`.

## Item Sprite Style

- Treat item icons as native 32x32 pixel art. If an AI or high-resolution draft is used, always finish with direct 32x32 cleanup; do not rely on automatic downscaling for final edges, layer order, or readability.
- For style references, prefer vanilla Isaac collectible sprites from `extracted_resources/resources/gfx/items/collectibles` and high-quality local mod references such as Epiphany, Astrobirth, and Auri only for general production traits. Do not copy, trace, or recreate another mod's specific item artwork.
- Match the common Isaac item language: compact readable silhouette, slightly handmade asymmetry, transparent corners, near-black 1-2 px outline, top-left lighting, muted grimy shadow colors, and small clusters of bright highlight pixels.
- Keep each material on a limited palette. A good default is outline, dark shadow, midtone, and one sparse highlight; use extra highlight colors only for the focal point.
- Avoid muddy antialiasing, soft gradients, airbrushed glow, photorealistic detail, and thin hairline elements that disappear at 32px. Favor hard pixel clusters and clear shape separation.
- Check layer order manually after scaling or cleanup. Foreground tools, blades, handles, knots, and cut points should visibly overlap in the intended order at 32px.
- For AI sprite prompts, specify final native 32x32 readability, hard pixel edges, limited palette, transparent padding, no text/UI/frame/pedestal, and "reference style traits only, no copying" when using vanilla or mod art as inspiration.

## Callback Guidelines

- Avoid adding `MC_POST_UPDATE`, `MC_POST_RENDER`, `MC_POST_PEFFECT_UPDATE`, or broad entity callbacks unless the item needs them continuously.
- If a broad callback is necessary, return early before expensive work. Check ownership, subtype, variant, or custom `GetData()` flags first.
- For item behavior, prefer `ItemData.callbacks` plus `CallbackManager` over scattered direct `AddCallback` calls.
- `scripts/callback_manager.lua` is an ItemData registrar, not a generic entity/room callback framework: it iterates only `ConchBlessing.ItemData`, registers after the item loader has loaded every behavior module (with `MC_POST_GAME_STARTED` retained only as an idempotent fallback), and only auto-filters familiar variants. Do not route a new entity or room subsystem through it without redesigning and verifying those semantics.
- CallbackManager keys are conveniences, not API contracts. Confirm the mapped `ModCallbacks` value, supported filter parameter, callback phase, arguments, and return behavior in both `callback_manager.lua` and the current docs before implementing a callback.
- The vendored IsaacScript Common root does not re-export `CallbackPriority`; `isc.CallbackPriority` is `nil` even though the vendor uses that enum internally. For direct `AddPriorityCallback` registrations, resolve the engine/REPENTOGON global `CallbackPriority` and use the documented `EARLY`/`LATE` values with a local numeric fallback. A room/entity loader is wrapped by core `pcall`, so an initialization error can leave a partially populated global namespace; capability flags must be assigned only after all required callbacks register, and `ConchBlessing.printError` must mirror failures to `Isaac.DebugString` so the runtime log exposes the original load error.
- Do not use the legacy ItemData `pickup` key as a collectible-acquisition event. It maps to `MC_POST_PICKUP_INIT`, whose callback entity and variant filter do not represent the player acquiring the collectible; use a documented add/remove-collectible callback with its exact signature or a cache-owned state transition instead.
- Match documented callback argument order exactly. Never hide a shifted or missing player argument by falling back to `Isaac.GetPlayer(0)`; reject invalid callback input and preserve the actual triggering player in multiplayer.
- Do not register empty callbacks through `ItemData.callbacks`.
- Cache static `ItemConfig` scans and repeated ID lists. Do not scan `CollectibleType.NUM_COLLECTIBLES` every frame.
- Avoid repeated `Isaac.GetRoomEntities()` or `Isaac.FindByType()` in per-entity update callbacks. Cache per-frame data or store runtime references when safe.
- Preserve multiplayer behavior by iterating `Game():GetNumPlayers()` when the effect can belong to any player.
- For attack-count thresholds, never group projectile callbacks by game frame. Reuse `scripts/lib/weapon_attack_tracker.lua` with the engine's semantic attack event. Breath and Dragon use REPENTOGON `MC_POST_TRIGGER_WEAPON_FIRED`: advance once per callback, never by `FireAmount` or `Weapon:GetNumFired`, and attribute only a direct `EntityPlayer` weapon owner. Multishot and familiar-owned mirror weapons must not add counts, while item-spawned secondary entities must not re-enter the counter. The base API has no exact fallback, and familiar-only weapon attacks have no shared player-action token; keep those dependency and scope limits explicit rather than restoring frame inference or item/familiar special cases.
- Prefer semantic callbacks and explicit state transitions over frame numbers or fixed frame delays. Do not infer game initialization, pickup completion, room readiness, or session phase from `Game():GetFrameCount()` when an ownership, pickup, room, entity, or queue event can establish it directly.
- `MC_POST_NEW_ROOM` and `MC_POST_NEW_LEVEL` can run before `MC_POST_GAME_STARTED` while SaveManager still exposes the previous on-disk run table. A room subsystem with run-scoped journals must keep an explicit lifecycle gate closed until its own game-start reconciliation finishes, close it on `MC_PRE_GAME_EXIT`, and make pre-start room/level/update/acquisition callbacks no-ops; never infer readiness from frame count.
- If a frame delay is unavoidable because the engine exposes no readiness event, document the reason, bound the wait, and clear pending state when a game or room transition invalidates that work.

## Damage Triggers And Provenance

- A collision callback proves contact, not applied HP damage. Base `MC_ENTITY_TAKE_DMG` runs before damage and may still be cancelled or changed; when a proc contract requires damage that actually occurred, use REPENTOGON `MC_POST_ENTITY_TAKE_DMG` or make the weaker fallback and its limitation explicit.
- Classify damage by its actual attack entity, not by an adjacent effect or a negative list of known attacks. For REPENTOGON post-damage callbacks, inspect `ExtraSource.Entity` first and fall back to `Source.Entity`; `ExtraSource` preserves laser, knife, and other hitbox provenance that the normal source may collapse to a player.
- Existing engine `DamageFlag` values may be bitwise-ORed and passed when code directly invokes a damage API such as `Entity:TakeDamage`, `Game:BombDamage`, or `Game:BombExplosionEffects`. Native spawned attacks normally receive their damage flags from the engine; do not assume an entity's `Flags` field contains `DamageFlag` values (`EntityBomb.Flags`, for example, contains `TearFlags`).
- There is no `DAMAGE_TEAR` and no safe project-owned `DamageFlag` or `TearFlags` bit allocation. Reuse `DamageProvenance.getEligiblePlayerAttack(source, extraSource, procKey)` instead of classifying attacks by flags or entity-type lists. When ownership and a discrete attack identity can be established, hit-triggered effects accept every player-owned damage source, including direct weapons, familiar-fired attacks, and attacks created by other items; fail closed when no player owner, actual source, or semantic attack boundary can be resolved.
- Prevent recursion with per-proc causal lineage, not a global attack-type or secondary-attack ban. Give each hit proc a stable key, carry the incoming provenance snapshot into its output through `markTriggeredAttack`, and accumulate that key in the lineage. A consumer rejects only a lineage that already contains its own key: the same laser, void, explosion, or other attack form from an unrelated source remains eligible, each distinct proc key may enter a causal branch once, and cycles terminate when a key would repeat.
- Treat recursion lineage and repeated-hit consumption as separate state. For a chance proc that should roll once per physical attack, call `DamageProvenance.tryClaimAttackProc(attackEntity, procKey)` after confirmed-damage, target, ownership, and item checks but before RNG; a failed roll still consumes that attack's opportunity. Claims are per exact attack entity and proc key, are not inherited through `procChain`, and must never be keyed by frame, damage cooldown, or target. A piercing tear, bomb explosion, Tech X ring, or Brimstone beam therefore rolls each proc once across all of its hits and targets, while distinct multishot entities remain distinct attacks.
- When the engine reuses a melee weapon's main `EntityKnife` across attacks, start a fresh claim state only from the semantic REPENTOGON `MC_POST_TRIGGER_WEAPON_FIRED` event and capability-checked `Weapon:GetMainEntity`, after its player ownership matches the callback owner; do not infer a reset from animation frames or elapsed time. Do not apply this generic reset to fresh or still-live tears, bombs, effects, or lasers: their exact entity is already the attack identity, and resetting an active laser can reopen an older Tech X ring or continuous beam. Canonicalize REPENTOGON melee hitboxes through `EntityKnife:GetHitboxParentKnife` for both claims and lineage resolution. Any other reused entity needs an exact producer callback/token with a documented lifecycle or must fail closed. A familiar body is a long-lived owner/contact hitbox, not a discrete attack, unless an exact producer explicitly opens its attack state. Do not treat `EntityFlag.FLAG_THROWN` as a throw-lifetime token or assume attack damage occurs synchronously between semantic pre/post callbacks unless the target runtime/source proves those contracts. If the engine exposes only a bare player/source, a source-less status tick, or another long-lived contact entity with no semantic attack boundary, use an explicit producer-scoped attack state or fail closed rather than permanently claiming the owner or inventing an attack token.
- Preserve lineage across delayed work by storing the plain provenance snapshot in the pending runtime record, and use `withTriggeredSource` around synchronous direct-damage APIs whose `Source` collapses to a player. Mark outputs at their semantic spawn/damage call, then let the shared tear, bomb, laser, knife, effect, projectile, familiar, and split-child inheritance callbacks propagate the lineage. Do not add item-name exclusions to consumers.
- Some engine-owned delayed damage, such as a status tick, may discard the applying attack and player from both `Source` and `ExtraSource`. Do not recover ownership from `DamageFlags`, a recent-hit/frame association, or the nearest player; fail closed unless a semantic source or a producer-carried provenance snapshot is available.
- When both an applied-damage callback and a base collision fallback are registered, the fallback must return before dedupe, lockout, RNG, or other state changes whenever `MC_POST_ENTITY_TAKE_DMG` is available. The fallback must also reject proc-ineligible provenance; its collision-only limitation remains part of the REPENTOGON-recommended compatibility path.
- Do not create a generic damage-flag setter merely to wrap bitwise OR. Do not extend `scripts/lib/damage_utils.lua` as a provenance registry because it is a legacy self-damage classifier and `ConchBlessing.DamageUtils` may be replaced by StatsAPI's implementation. Do not use `DAMAGE_CLONES` as a generic origin marker; reserve it for preventing recursive damage re-entry when code deliberately calls `TakeDamage` again.

## Save Data

- Use `ConchBlessing.SaveManager` for persistent and run-scoped state.
- When a system grants ordinary collectible copies that must later be removed while identical player-owned copies may coexist, inventory totals are not ownership. Persist the active system-owned contribution and the player's pre-grant baseline, update them together, and remove only copies above that baseline. If a cap means “ever during this ownership” rather than “currently active,” persist that lifetime total separately. Chronus familiar conversions use `itemGrants`, `itemGrantBaselines`, and `itemGrantTotals` for these roles.
- Choose the narrowest matching scope: `GetRunSave`, `GetFloorSave`, `GetRoomSave`, or `GetTempSave`. Pass the actual player/entity when state has an owner, and use `noHourglass` only when reversal is intentionally suppressed.
- Use `GetPersistentSave` for non-setting data that must survive across runs, such as project-managed unlock state, and `GetSettingsSave` for configuration. These file scopes do not have matching `TryGet*` forms; consult the vendored annotations instead of inferring one.
- In multiplayer, bind state to the callback's triggering player or SaveManager's entity-scoped save. Do not use player 0 or a module-global table as a substitute for ownership unless the design is explicitly shared across all players.
- Store persistent MCM/settings data under `SaveManager.GetSettingsSave().config` through the existing config helpers.
- Store serializable data only. Do not write entity, sprite, font, RNG, or userdata objects into saved tables.
- If runtime entity references are cached, clear them on room/game transitions and keep them out of SaveManager data.
- The gallery manager owns the shared `GetRunSave(nil, false).appraisalGallerySession` record. Its frozen catalog, floor identity, serializable native graph manifest, per-room `closedRooms`, fixed stock slot/`InitSeed` identities, and visit/acquisition journals form a floor-scoped ledger even though the physical table is run save. The engine owns physical native `RoomDescriptor` and room-entity persistence; the manager rebinds them to the saved manifest and never stores userdata. Every same-floor normal return, unexpected native exit, recovery escape, continue, and repeated use must preserve the ledger as a completed tombstone. A genuinely different floor resets it only after any unresolved entry/acquisition journal has reached a durable cross-floor settlement tombstone; never erase an applied reward journal merely because `MC_POST_NEW_LEVEL` fired. Per-visit origin, owner, payment, pending reward, and transition fields reset for each paid entry. Keep only serializable identity, exact held IDs, plain smelt-count baselines/deltas, commit progress, phase, and transition metadata there; StageAPI owns only its optional Atropos custom door, not the native room graph or stock.
- Atropos owns `atroposNativeDeathCertificate` for its native-dimension session/choice ledger and `atroposChoiceOptionGroups` for temporary option-link ownership. The latter stores only room identity and pickup `InitSeed` to original nonzero `OptionsPickupIndex` mappings; remove a room's mappings when the last owner restores/relinquishes them so stale records cannot overwrite another provider.
- A call to vendored `SaveManager.Save()` returning without an exception is not proof of persistence: that implementation silently returns on initialization, copy, backup, or JSON-validation failures. Write-ahead gameplay transactions must observe the matching `POST_DATA_SAVE` completion callback before performing their irreversible step. After `SaveManager.Init`, public `SaveManager.SaveCallbacks.*` values already include the mod-specific prefix; register those IDs directly on `ConchBlessing.originalMod` and do not prepend `__SAVEMANAGER_UNIQUE_KEY` again. Like every callback registered through `Mod:AddCallback`, a SaveManager callback receives the registered mod as its first argument: `PRE_DATA_SAVE` handlers that use or return the payload must use `function(_, saveData)`, never mistake the mod object for serializable data.
- Once a transaction is committed, never infer that it rolled back solely because a later inventory or currency total equals a pre-transaction value; unrelated gains and spending can collide with that total. Revalidate a provider-owned semantic delta when one exists, otherwise trust the committed journal or fail closed without mutating unrelated state.
- Never reopen manager-owned stock after a failed automatic grant unless the exact transaction-owned inventory delta was removed, the complete post-rollback inventory snapshot exactly matches the prepared baseline, and a semantic rollback-complete state was durably saved. Persist cancellation-complete and rollback-complete phases before clearing their journals so a failed cleanup save or continue can finish safely. If rollback or every save barrier fails, retain the pending journal and route the session to recovery so the same stock cannot grant twice.
- SaveManager journal data and the engine's player inventory are separate persistence stores, so a hard process crash between their writes cannot be made truly atomic by Lua. Use prepared/complete phases and exact snapshots to fail closed, then cover quit/continue at each transaction boundary in the in-game verification matrix; do not invent frame-based reconciliation or guess ownership from coincidental totals.

## Verification

- There is no normal automated Lua test suite for this mod.
- For Python generator changes, run `python3 -m py_compile generate_xml.py`.
- After changing generated-content rules, run `python3 generate_xml.py` in a Python environment with Pillow and inspect `content/gfx/death_items.png`.
- For any generated XML change, parse or validate the XML, run the generator twice, and confirm the second run produces no diff. If entity generation changes, confirm both existing familiars and every new non-familiar entry survive.
- For Lua syntax checks, use the local Lua tooling only if available in the environment.
- For gameplay changes, verify in-game with debug mode off and then with debug mode on only when logs are needed.
- After XML, ANM2, STB, sound, or spritesheet changes, fully restart the game and inspect the game log for parse and missing-resource errors; a Lua-only reload is not sufficient verification.
- For a custom entity or monster, verify name-to-ID lookup and console spawning, every animation/event and AI state, collision/damage/death/removal, room exit/re-entry, continue behavior, and any friendly/charmed/frozen or multiplayer paths the design supports.
- For a custom effect, verify animation completion, timeout/removal, pause behavior, and every room/game transition that can invalidate it; confirm no orphan entity or permanently active broad callback remains.
- For an authored room, verify it in the room editor and through natural seeded generation in game. Check every supported shape/door, grid and entity placement, room clear and award behavior, revisit/continue, and that custom entity IDs match the runtime XML.
- For room placement or stage work, test normal and alternate stages, applicable dimensions, Curse of the Maze/Glowing Hourglass transitions, Greed Mode if supported, and compatibility without optional frameworks. If a framework becomes required, verify the missing-dependency failure path too.
