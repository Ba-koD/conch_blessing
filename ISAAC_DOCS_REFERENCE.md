# Isaac Docs Reference Map

Last checked: 2026-07-15

Current base docs (Repentance+): https://wofsauge.github.io/IsaacDocs/rep/

REPENTOGON docs: https://repentogon.com/docs.html

This file is a routing guide for Wofsauge's Binding of Isaac Lua API docs. Use it to decide which docs page to open before changing this mod.

## Basic Rules

- Use the `/rep/` docs first for the current Repentance+ base API. Search engines often return `/abp/` or `oldDocs`; those are legacy references unless the task explicitly targets old behavior.
- This repository does not currently pin a minimum Repentance/Repentance+ executable. A method in `/rep/` is not proof that it exists on older supported builds, and a method in editor stubs is not proof that it is base-game API. Check the intended runtime and keep capability guards where the baseline is not guaranteed.
- Until a minimum build is pinned, a new API choice must either declare and test its target build or provide a capability-checked fallback. If neither is possible, resolve the support/dependency policy before writing the feature.
- Use `repentogon.com` for REPENTOGON-only classes, callbacks, and method extensions. Use the installed provider's versioned documentation/source for StatsAPI, Magic Conch, MCM, EID, StageAPI, and other mod APIs.
- For behavior questions, read in this order: class/function page, related enum page, related XML page, then tutorial or FAQ page.
- For callbacks, always read the callback signature, optional parameter matching, and return-value behavior before returning anything from a callback.
- For XML, read the exact file page before choosing `content` or `resources`; add/merge/replace/no-effect behavior differs by XML file.
- For item content, remember the local source of truth: item metadata starts in `scripts/conch_blessing_items.lua`, then `generate_xml.py` may regenerate `content/items.xml` and `content/itempools.xml`.
- For custom entities, remember the current local limitation: `generate_xml.py` owns all of `content/entities2.xml` but emits familiar entries only. It is not safe to add a manual monster entry until the source/generator workflow is extended.
- The docs are community-maintained. If documented behavior differs from a reproducible test on the intended target build, trust that runtime result for that build. Record durable project-wide conclusions in `AGENTS.md` or this reference map, keep only narrow implementation quirks near the affected code, and do not generalize one installed build's result to an unpinned compatibility baseline.

## Local Workflow Notes

- New collectible or trinket: add or update `ConchBlessing.ItemData` in `scripts/conch_blessing_items.lua`, then check `items.xml`, `ItemConfig`, and the relevant callback docs. If metadata affects XML, check whether `generate_xml.py` should regenerate `content/items.xml` or `content/itempools.xml`.
- New passive stat effect: set the item `cache` data, implement `MC_EVALUATE_CACHE`, and change player stat fields only inside cache evaluation.
- New active item effect: check `MC_USE_ITEM`, `MC_PRE_USE_ITEM`, `UseFlag`, `ActiveSlot`, and `items.xml` charge fields before choosing callback behavior.
- Active-item callbacks registered through CallbackManager still receive the normal mod-callback arguments. Keep the leading mod/self argument in the Lua function and use the documented triggering-player position; never recover a shifted signature with `Isaac.GetPlayer(0)`.
- New familiar item: check `EntityPlayer:CheckFamiliar`, `CacheFlag.CACHE_FAMILIARS`, `EntityFamiliar`, and `MC_FAMILIAR_*` callbacks.
- Attack-count threshold: reuse `scripts/lib/weapon_attack_tracker.lua` with REPENTOGON `MC_POST_TRIGGER_WEAPON_FIRED`. Count each direct player-owned weapon trigger once, regardless of `FireAmount`; reject familiar owners instead of folding projectile creation by frame. There is no exact base fallback. Test Breath and Dragon with multishot, Incubus/Twisted Pair, familiar-only Lilith as the documented unsupported scope, charged and continuous weapons, item-spawned secondary entities, multiplayer, room resets, and the no-REPENTOGON registration path.
- Proc after actual attack damage: reuse `scripts/lib/damage_provenance.lua` with REPENTOGON `MC_POST_ENTITY_TAKE_DMG` and call `getEligiblePlayerAttack(source, extraSource, procKey)`. Accept every source whose player ownership can be resolved, then call `tryClaimAttackProc(attackEntity, procKey)` before RNG when the contract is one roll per physical attack; failed rolls consume the opportunity, and claims do not inherit to newly-created child attacks. Carry the returned provenance snapshot into `markTriggeredAttack` or `withTriggeredSource`, and store the snapshot with delayed work. Reject only the current proc's lineage key; damage flags cannot serve as a project-owned provenance namespace. Reusable melee `EntityKnife` main entities obtain a fresh claim state from `MC_POST_TRIGGER_WEAPON_FIRED` plus `Weapon:GetMainEntity` after owner matching, never a frame/delay reset; their child hitboxes resolve both claims and lineage through `GetHitboxParentKnife`. Do not reopen live laser/entity claims generically. Any other reused entity, including a familiar body used as a contact hitbox, requires an exact semantic producer callback/token with a proven lifecycle or fails closed. `FLAG_THROWN` alone and undocumented pre/post damage ordering are not sufficient lifecycle proof. If engine-owned delayed damage has discarded both source and owner or only a bare unscoped player remains, fail closed rather than associating it with a recent frame or nearby player. If a base collision fallback is retained, make it a no-op whenever the post-damage hook exists, restrict it to the sources it can identify reliably, use the same attack claim, reject the same lineage there, and document that collision cannot prove final applied damage.
- Unavoidable-but-revivable player death: the current project path is direct `Entity:Kill()` rather than damage. Before treating a new effect as complete, test no protection, shields, active invincibility, ordinary and conditional extra lives, no-extra-life game over, and multiplayer ownership. Do not call `WillPlayerRevive`/`Revive` to consume or reproduce the engine's revival flow manually.
- New spawn or morph behavior: check `Isaac.Spawn`, `Game():Spawn`, `EntityPickup:Morph`, `Room:FindFreePickupSpawnPosition`, and the `EntityType` / variant enum pages.
- New room or floor transition behavior: prefer `MC_POST_NEW_ROOM`, `MC_POST_NEW_LEVEL`, `MC_POST_GAME_STARTED`, or `MC_PRE_GAME_EXIT` before considering broad per-frame callbacks.
- Run-scoped room journals: base callback ordering allows `MC_POST_NEW_ROOM`/`MC_POST_NEW_LEVEL` before `MC_POST_GAME_STARTED`. Keep a semantic ready gate closed until game-start reconciles the saved run, close it on pre-exit, and arm any `StartRoomTransition` issue latch before the call so a synchronous/no-op callback cannot recursively reopen it.
- New REPENTOGON native special-room graph: use `scripts/rooms/native_gallery_rooms.lua` only as a placement/reference pattern and define a separate graph, ownership, and save contract. Engine dimension 2 is shared with Death Certificate, so check for occupancy before mutation, persist plain descriptor identity, reveal the native descriptors deliberately, and test mixed room shapes, vanilla minimap visibility, continue, repeated same-floor use, and foreign-room conflicts.
- New StageAPI extra-room system: do not treat Appraisal as a LevelMap example; its room graph is native REPENTOGON placement and StageAPI only supplies the Atropos return door. Select and document a separate LevelMap/LevelRoom architecture when a different feature genuinely needs StageAPI extra rooms.
- Persistent room state: obtain the relevant descriptor `ListIndex` and pass it to `SaveManager.GetRoomSave`/`TryGetRoomSave`; choose `noHourglass` deliberately. Use `GetTempSave` only for state that should be discarded on leaving the room.
- New custom entity or monster: do not put it in ItemData or hand-edit the generated `content/entities2.xml`. First add a dedicated entity metadata source and extend/replace the generator workflow, then wire an explicit entity loader into core. Cache `Isaac.GetEntityTypeByName` and `Isaac.GetEntityVariantByName`; failed lookup returns `0` and `-1` respectively, and there is no subtype-by-name API.
- New NPC behavior: read the NPC init/update/render/death/collision callback notes together. Filter by callback type parameter and then variant/subtype, use `EntityNPC.Pathfinder` and animation events where applicable, and distinguish kill, completed NPC death, removal, and room-unload cleanup.
- New room pack: author only the additional layouts with a Repentance-compatible editor, export STB files to `content/rooms` using the exact vanilla stage filenames, and compare with current extracted room files. `resources/rooms` replaces room sets and is not the default additive path.
- New room-layout substitution: use `MC_PRE_ROOM_ENTITY_SPAWN` only to replace entities already being spawned from a layout. It does not author, register, or place a room.
- New full custom floor or dynamic room placement: stop and select an explicit, versioned StageAPI or REPENTOGON architecture first. Do not use `MC_LEVEL_GENERATOR`, `content/stages.xml`, or an undeclared locally installed framework as an implicit dependency.
- New ANM2-backed content: treat animation names, event names, layer/null names, and spritesheet paths as code-facing contracts. Update Lua, XML, ANM2, and spritesheets together and fully restart the game.
- Save data: use `ConchBlessing.SaveManager` first. Use the Isaac save-data docs only to understand the underlying `Isaac.SaveModData` / `Isaac.LoadModData` behavior.
- Runtime entity state: use `Entity:GetData()` for temporary per-entity state under a unique `__ConchBlessing...` key, but never store entity userdata, sprite objects, RNG objects, or other userdata inside SaveManager tables.
- Provider API change: begin at the matching row in **Project Integration Entry Points**, then inspect the installed provider version. A sibling mod merely being installed does not make it a declared dependency or authorize copying its implementation.

## Search Tips

- Start with docs search or a web query like `site:wofsauge.github.io/IsaacDocs/rep EntityPlayer AddCacheFlags`.
- For an API that might be REPENTOGON-only, search both `site:wofsauge.github.io/IsaacDocs/rep MethodName` and `site:repentogon.com MethodName`; then check the intended minimum runtime rather than trusting autocomplete.
- On large class pages, use browser search for the exact API name, for example `GetCollectibleNum`. The visible table of contents may split method words with dots, but the real API name is still the plain Lua method name.
- If a function signature links to enum types, open those enum pages immediately. Enum details often explain names used in XML or flags used by callbacks.
- If a page has `Bugs`, `Notes`, or `Version Difference` sections, treat those as part of the API contract for this mod.
- If a callback has optional entity-type or variant filtering, use it instead of checking every entity inside a broad callback.

## Common Cautions

- `/rep/` is the current Repentance+ documentation, not an archived minimum-Repentance contract. Keep fallbacks for older builds until the project explicitly drops them.
- `Isaac.GetFrameCount()` and `Game():GetFrameCount()` are not interchangeable. Check the relevant page before using frame counts for timers.
- `Isaac.GetRoomEntities()` and `Isaac.FindByType()` are useful, but avoid repeated scans inside broad update callbacks unless the result is cached or the entity set is small.
- `MC_LEVEL_GENERATOR` exists in the enum and local callback mapping but is documented as inactive; never use it as a custom-floor hook.
- `MC_POST_ENTITY_KILL`, `MC_POST_NPC_DEATH`, and `MC_POST_ENTITY_REMOVE` are different lifecycle events. Removal can occur during room unload or run shutdown and is not proof of a death.
- `MC_PRE_ROOM_ENTITY_SPAWN` can replace layout spawn entries; it is not a room registration or room placement API.
- Entity `Type`, `Variant`, and `SubType`, room type/variant/subtype, `LevelStage`, `StageType`, STB stage IDs, and entity `bossID` are separate identifier domains. Do not reuse or compare them without checking the relevant docs.
- Native grid entities can be spawned and modified, but adding a brand-new grid-entity type requires an explicitly adopted extender/framework mechanism or a normal entity/effect emulation.
- `content/entities2.xml` is currently generator-owned and familiar-only. A manual monster or boss entry will be lost on the next generator or pre-push run.
- XML/ANM2/STB/sound/spritesheet changes require a full game restart; `luamod` only proves Lua reload behavior.
- `items_metadata.xml` is usually not needed for this mod; quality and tags should generally live in `items.xml` data generated from `ConchBlessing.ItemData`.
- `itempools.xml` documentation is thin. For pool edits, compare against existing generated `content/itempools.xml`, the `ItemPoolType` enum, and in-game behavior.
- Search results can mix Repentance+, AB+, and old docs. Verify the URL contains `/rep/` before relying on a page.
